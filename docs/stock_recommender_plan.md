# 주간 종목 추천기 개발 계획

## 1. 문서 목적

현재 AutoTrade는 사용자가 `AUTOTRADE_TARGET_SYMBOLS`를 직접 넣어 자동매매 대상을 확정한다.
다음 단계에서는 이 과정을 아래의 주간 추천 파이프라인으로 바꾼다.

1. 코드로 재현 가능한 규칙으로 주간 후보 20개를 선정한다.
2. 결과를 CSV와 Markdown 리포트로 저장한다.
3. Codex는 리포트를 입력받아 상위 3~4개 후보의 장단점과 제외 리스크를 해석한다.
4. 사용자가 승인한 3~4개만 자동매매 본체의 실제 대상 종목으로 사용한다.

이 문서는 기능 구현 전에 범위, 모듈 경계, 단계별 개발 계획, 리스크, 검증 방식을 고정하기 위한 planner/manager 문서다.

## 2. 목표 흐름

### 2.1 목표 사용자 흐름

1. 주말 또는 주간 마지막 거래일 장마감 후 추천기 프로그램을 실행한다.
2. 추천기는 기준 유니버스에서 필터, 점수 계산, 상위 N개 선정, 제외 규칙을 적용해 후보 20개를 만든다.
3. 추천기는 후보 목록과 점수 구성요소를 CSV/Markdown/JSON으로 저장한다.
4. 사용자는 Markdown 리포트를 Codex에 넣어 상위 3~4개 추천과 제외 사유 요약을 받는다.
5. 사용자는 승인된 종목만 별도 승인 산출물에 기록한다.
6. 자동매매 실행기는 승인 산출물을 우선 사용하고, 없으면 기존 `AUTOTRADE_TARGET_SYMBOLS`로 fallback 한다.

### 2.2 첫 버전 비목표

- 뉴스, 공시, 재무제표, 텍스트 감성 같은 비결정적 신호를 점수 계산에 넣지 않는다.
- Codex가 점수 계산을 직접 수행하거나 자동으로 종목을 확정하게 하지 않는다.
- 승인 결과를 바로 `.env`에 덮어쓰지 않는다.
- 처음부터 전체 KRX 전 종목 크롤링과 완전 자동 메타데이터 동기화를 목표로 하지 않는다.

## 3. 설계 원칙

- 추천 로직은 같은 입력과 파라미터면 항상 같은 결과를 내야 한다.
- 데이터 수집, 점수 계산, 리포트 출력, 승인 반영을 분리한다.
- 추천 결과와 최종 승인 결과를 모두 파일로 남겨 추적 가능하게 한다.
- Codex는 해석, 비교, 요약만 담당하고 최종 승인 권한은 사용자에게 남긴다.
- 기존 자동매매 실행 경로를 바로 깨지 않도록 승인 산출물 우선, 환경변수 fallback 구조를 유지한다.
- 첫 버전은 좁고 안정적인 유니버스부터 시작하고 이후에 확장한다.

## 4. 권장 모듈 경계

현재 `strategy/selector.py`는 전략 종류 선택 팩토리 역할을 맡고 있으므로, 주간 종목 추천 로직을 여기에 섞으면 책임이 모호해진다.
추천기는 전략 신호 계산과 다른 단계이므로 별도 경계로 분리하는 편이 안전하다.

### 4.1 권장 신규 경계

- `src/autotrade/recommendation/`
  - 유니버스 필터
  - 점수 계산
  - 상위 N개 선정
  - 제외 규칙
  - 추천 결과 데이터 모델

### 4.2 기존 모듈과의 연결

- `config/`
  - 기존 `AUTOTRADE_TARGET_SYMBOLS`는 유지한다.
  - 추후 승인 산출물 경로 같은 선택 설정을 추가할 수 있다.
- `data/`
  - 추천기에 필요한 바 데이터, 유니버스 메타데이터, 정합성 검증을 제공한다.
- `report/`
  - CSV/Markdown/JSON 결과물 저장과 렌더링을 담당한다.
- `runtime/`
  - 승인된 종목을 실제 실행 대상 종목으로 해석하는 경계만 맡긴다.
- `tools/` 또는 `cli`
  - 주간 추천 리포트 생성과 승인 결과 반영용 진입 명령을 제공한다.

### 4.3 현재 코드 기준 주의점

- `runtime/live_cycle.py`는 여러 심볼 순회를 이미 지원하지만, 설정 기본값은 여전히 `settings.target_symbols`다.
- `runtime/operation_services.py`의 paper broker 초기화는 현재 `settings.target_symbols[0]` 기준 주문가능현금을 조회한다.
- `broker/smoke.py`도 첫 번째 심볼을 기준으로 현재가와 주문가능수량을 점검한다.

즉, 승인 종목 3~4개 연동은 단순히 추천기만 추가하는 문제가 아니라, 일부 운영 경로에 남아 있는 단일 대표 심볼 가정을 함께 점검해야 한다.

## 5. 첫 버전 입력과 출력

### 5.1 입력

- 기준 유니버스 파일
  - 첫 버전은 전체 시장 자동수집보다 관리 가능한 seed universe 파일을 권장한다.
  - 컬럼 예시: `symbol`, `name`, `asset_type`, `sector`, `is_etf`, `is_inverse`, `is_leveraged`, `active`
- 바 데이터
  - 일봉 기준 최소 120영업일 이상
- 추천 파라미터
  - 최소 거래대금 기준
  - 모멘텀 윈도우
  - 변동성 상한 또는 패널티
  - 추세 지표 윈도우
  - 업종/ETF 허용 또는 제외 규칙
  - 최종 후보 수와 승인 목표 수

### 5.2 출력

- `weekly_candidates_YYYYMMDD.csv`
- `weekly_candidates_YYYYMMDD.md`
- `weekly_candidates_YYYYMMDD.json`
- `approved_symbols_YYYYMMDD.json` 또는 이에 준하는 승인 산출물

Markdown 리포트에는 최소한 아래 정보가 들어가야 한다.

- 생성 시각과 기준 거래일
- 사용한 유니버스 크기
- 필터 통과 수와 탈락 사유 요약
- 최종 후보 20개 테이블
- 점수 총점과 구성요소 점수
- 하드 제외 규칙에 걸린 종목 요약
- Codex에 바로 붙여 넣을 수 있는 검토 프롬프트 초안

## 6. 추천 로직 초안

첫 버전은 아래 4단계로 고정한다.

### 6.1 유니버스 필터

- 비활성 종목 제외
- 메타데이터 불완전 종목 제외
- 최소 바 수 미달 종목 제외
- 최소 평균 거래대금 미달 종목 제외
- 레버리지/인버스 ETF 제외
- 수동 제외 업종 또는 수동 제외 심볼 제거

### 6.2 점수 계산

각 점수는 0~100 범위로 정규화하고 가중합으로 총점을 만든다.

- 거래대금 점수
  - 최근 20일 평균 거래대금 percentile
- 모멘텀 점수
  - 20일, 60일, 120일 수익률 조합
- 변동성 점수
  - 최근 20일 또는 60일 실현변동성 기반 패널티 또는 적정 범위 점수
- 추세 점수
  - 가격과 이동평균 정배열 여부
  - 이동평균 기울기

가중치는 설정으로 분리하되 첫 버전 기본값은 문서에 고정한다.

### 6.3 상위 N개 선정

- 총점 내림차순 정렬
- 동점이면 거래대금, 모멘텀, 심볼 코드 순으로 tie-break
- 기본 후보 수는 20개

### 6.4 제외 규칙

- 동일 섹터 쏠림 상한
- 동일 기초지수 또는 유사 ETF 중복 상한
- 변동성 과다 종목 제외
- 거래대금 급감 종목 제외
- 최근 데이터 결손 종목 제외

첫 버전에서는 설명 가능한 하드 룰만 넣고, 해석이 필요한 리스크는 리포트에 남겨 Codex가 비교/설명하도록 한다.

## 7. Codex 역할 정의

Codex는 추천기 점수를 대신 계산하지 않는다.
Codex는 추천기가 만든 산출물을 읽고 아래 역할만 수행한다.

- 상위 후보 20개 중 3~4개 압축 제안
- 왜 3개 또는 4개가 적절한지 비교 설명
- 제외해야 할 종목의 리스크 요약
- 섹터/ETF 중복, 변동성 편향, 거래대금 편향 같은 포트폴리오 관점 경고

즉, 추천기는 결정적 엔진이고 Codex는 해석 레이어다.

## 8. 승인 반영 방식

첫 버전에서는 승인 결과를 `.env`에 직접 덮어쓰지 않는 것이 안전하다.

### 8.1 권장 흐름

1. 추천기가 후보 리포트를 만든다.
2. 사용자가 Codex 검토를 거쳐 승인 종목 3~4개를 확정한다.
3. 별도 승인 산출물에 저장한다.
4. 자동매매 실행기는 승인 산출물이 있으면 그것을 우선 사용한다.
5. 승인 산출물이 없으면 기존 `AUTOTRADE_TARGET_SYMBOLS`를 사용한다.

### 8.2 이유

- 운영 중 `.env` 자동 수정은 추적과 복구가 어렵다.
- 승인 결과를 파일로 남기면 주차별 변경 이력을 비교할 수 있다.
- fallback 이 있으면 추천기 기능 도입 전후를 안전하게 공존시킬 수 있다.

## 9. 단계별 구현 계획

### 9.1 1단계 - 데이터 계약 고정

### 작업 요약
- 추천기 입력 메타데이터와 출력 결과 모델을 고정한다.

### 예상 영향 파일/모듈
- `docs/stock_recommender_plan.md`
- `docs/roadmap.md`
- 추후 `src/autotrade/recommendation/models.py`
- 추후 `tests/unit/recommendation/`

### 구현 단계
- 유니버스 메타데이터 스키마 확정
- 추천 결과 모델과 파일 포맷 확정
- 승인 산출물 포맷 확정

### 리스크 포인트
- ETF/업종 메타데이터 품질이 낮으면 필터 결과가 흔들릴 수 있다.
- 승인 산출물 포맷이 늦게 바뀌면 runtime 연동 범위가 커진다.

### 검증 계획
- 샘플 메타데이터와 샘플 추천 결과를 문서 예시로 검토

### 9.2 2단계 - 결정적 추천 엔진 구현

### 작업 요약
- 필터, 점수 계산, 랭킹, 제외 규칙을 순수 로직으로 구현한다.

### 예상 영향 파일/모듈
- 추후 `src/autotrade/recommendation/filters.py`
- 추후 `src/autotrade/recommendation/scoring.py`
- 추후 `src/autotrade/recommendation/selection.py`
- 추후 `tests/unit/recommendation/`

### 구현 단계
- 유니버스 필터 구현
- 특성값 계산과 점수 정규화 구현
- 가중합과 tie-break 규칙 구현
- 제외 규칙과 최종 20개 선정 구현

### 리스크 포인트
- 작은 유니버스에서는 percentile 점수 왜곡이 생길 수 있다.
- 점수 구성요소가 많아질수록 설명 가능성이 떨어질 수 있다.

### 검증 계획
- 입력 fixture 기준 고정 순위 테스트
- 필터 탈락 사유 테스트
- 동점 정렬과 제외 규칙 회귀 테스트

### 9.3 3단계 - 리포트 출력 구현

### 작업 요약
- CSV/Markdown/JSON 산출물을 작성하고 주간 아카이브 구조를 고정한다.

### 예상 영향 파일/모듈
- 추후 `src/autotrade/report/`
- 추후 `src/autotrade/recommendation/reporting.py`
- 추후 `tests/unit/report/`

### 구현 단계
- CSV writer
- Markdown renderer
- JSON archive writer
- Codex 검토용 프롬프트 섹션 생성

### 리스크 포인트
- 리포트가 너무 장황하면 Codex 입력 품질이 떨어진다.
- CSV/Markdown 간 정렬 기준 불일치가 생길 수 있다.

### 검증 계획
- golden text 테스트
- CSV 컬럼 순서 고정 테스트

### 9.4 4단계 - 승인 결과 연결

### 작업 요약
- 승인된 3~4개 종목을 runtime 이 읽을 수 있는 경로를 만든다.

### 예상 영향 파일/모듈
- 추후 `src/autotrade/config/`
- 추후 `src/autotrade/runtime/`
- 추후 `src/autotrade/cli.py`
- 추후 `tests/unit/config/`
- 추후 `tests/unit/runtime/`

### 구현 단계
- 승인 산출물 로더 추가
- approved symbols 우선, env fallback 해석 규칙 추가
- 승인 결과를 확인하는 소규모 CLI 또는 도구 추가

### 리스크 포인트
- 승인 산출물 부재 또는 손상 시 운영이 멈추지 않도록 fallback 이 필요하다.
- 승인 종목 수와 리스크 설정의 최대 보유 수가 다를 수 있다.
- 일부 운영 경로가 첫 번째 심볼을 대표 심볼처럼 취급하므로, 승인 종목 파일 연동 시 단일 심볼 가정이 남지 않도록 점검해야 한다.

### 검증 계획
- 승인 산출물 우선 적용 테스트
- fallback 테스트
- 손상 파일 복구 또는 무시 정책 테스트

### 9.5 5단계 - 운영 연결

### 작업 요약
- 주간 리뷰와 추천기 실행 타이밍을 운영 플로우에 연결한다.

### 예상 영향 파일/모듈
- 추후 `src/autotrade/runtime/operations.py`
- 추후 `docs/operations.md`
- 추후 `tools/`

### 구현 단계
- 수동 CLI로 먼저 제공
- 안정화 후 `market_close` 또는 주간 배치와 연결 검토

### 리스크 포인트
- 주간 리뷰와 추천기까지 한 번에 묶으면 실패 범위가 커진다.
- 최초에는 수동 실행이 더 안전할 수 있다.

### 검증 계획
- 샘플 주간 데이터로 end-to-end dry run
- 기존 weekly-review 흐름과 충돌 없는지 확인

## 10. 우선 결정할 사항

- 첫 버전 유니버스를 전체 KRX가 아니라 seed universe 로 시작할지
- ETF와 개별주를 같은 점수 체계로 볼지, 자산군별 최소 quota 를 둘지
- 동일 섹터 최대 허용 개수를 몇 개로 둘지
- 승인 종목 기본 수를 3개로 둘지 4개 허용으로 둘지
- 추천 리포트 생성 시점을 금요일 장마감 직후로 둘지 주말 수동 배치로 둘지

## 11. 권장 첫 구현 범위

가장 안전한 첫 구현 범위는 아래다.

1. seed universe 파일 기반 추천기
2. 일봉 기반 필터/점수/랭킹/제외 규칙
3. 주간 후보 20개 CSV/Markdown/JSON 생성
4. 승인 종목 파일 저장
5. runtime 이 승인 종목 파일을 우선 읽고 없으면 기존 환경변수 사용

이 범위를 넘는 실시간 뉴스, 자동 승인, 전체 시장 자동 메타데이터 수집은 2차 단계로 미룬다.
