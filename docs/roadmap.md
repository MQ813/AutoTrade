# 국내 ETF 및 단일 종목 현물 자동매매 로드맵

## 1. 목적

개인 개발자 기준으로 국내 ETF/주식 현물 자동매매 시스템을 설계, 구현, 검증, 운영하기 위한 기준 문서다. 첫 목표는 고수익 전략이 아니라 안전하게 동작하는 MVP다.

전제:

- 시장: 국내 ETF 및 국내 주식 단일 종목 현물
- 주체: 개인 개발자
- 방식: 저빈도 자동매매
- 초기 목표: 소액 실전 가능한 안정적 1차 버전
- 우선순위: 전략 복잡도보다 시스템 안정성, 리스크 통제, 운영 가능성

## 2. 기본 방향

핵심 원칙:

1. 복잡한 전략으로 시작하지 않는다.
2. 레버리지/인버스/선물/초단타는 제외한다.
3. 전략 코드와 증권사 API 코드를 분리한다.
4. 백테스트 -> 모의투자 -> 소액 실전 -> 점진 증액 순서로 간다.
5. 수익률보다 장애 대응, 주문 정합성, 리스크 제한을 먼저 완성한다.

초기 권장 조합:

- 상품: 국내 ETF와 국내 주식 단일 종목
- 시간: 정규장만
- 빈도: 하루 1회 또는 30~60분 단위
- 보유: 최대 1~3개
- 주문: 지정가 위주
- 전략: 1개

## 3. 증권사

한국투자증권

## 4. 범위

포함:

- 국내 ETF/주식 현물
- 정규장 자동매매
- 일봉 또는 30분봉 전략
- 지정가 주문
- 단일 계좌, 단일 전략
- 장중 점검, 장마감 리포트

제외:

- 레버리지/인버스 ETF
- 선물/옵션, 초단타/HFT
- 뉴스 기반 자동매매
- 강화학습/딥러닝 전략
- 멀티 브로커
- 복수 전략 동시 운영

## 5. 유니버스 선정

초기에는 ETF와 국내 주식을 합쳐 3~10개만 관리한다.

공통 기준:

- 충분한 거래대금과 유동성
- 이해하기 쉬운 구조/사업
- 선정 사유 설명 가능
- 단일 이벤트에 과도하게 흔들리지 않음

ETF 기준:

- 광범위 지수형
- 괴리율/NAV 이슈가 과도하지 않음

국내 주식 기준:

- 충분한 시가총액과 유동성
- 테마주보다 업종 대표주 또는 실적 기반 종목
- 단일 공시/수급 이벤트 의존도 낮음

제외:

- 테마형 ETF
- 거래량 부족 ETF/종목
- 급격한 변동성 의존 종목
- 레버리지/인버스 ETF
- 관리종목, 투자주의/경고/위험 종목

예시 방향: 대형지수형 ETF, 기술/성장 대형지수형 ETF, 채권형/현금대용 ETF 일부, 코스피 대형주/업종 대표주, 유동성 안정 우량주.

## 6. 전략

첫 전략 조건:

- 설명 가능
- 단순한 룰
- 쉬운 백테스트
- 실전 매수/매도 근거 추적 가능

권장 유형:

- 일봉 추세추종: 이동평균 진입/이탈. 단순하고 검증 쉽지만 횡보장 약함.
- 30분봉 저빈도 추세: 반응은 빠르지만 체결/슬리피지 관리 필요.
- 변동성 필터 + 현금 비중: 낙폭 관리에 유리하나 강한 상승장 기회손실 가능.

비추천:

- 조건 과다 복합 전략
- 분 단위 초단타
- 뉴스/텍스트 기반
- AI 모델 강의존

## 7. 시스템 아키텍처

```text
AutoTrade/
├─ config/
├─ data/
├─ strategy/
├─ risk/
├─ broker/
├─ execution/
├─ portfolio/
├─ scheduler/
├─ report/
├─ logs/
└─ tests/
```

모듈 역할:

- `config`: 환경, 계좌/장운영/종목, API 키, 운영 파라미터.
- `data`: 시세 수집, 종목 목록, 일봉/분봉 저장, 정합성.
- `strategy`: 신호, 진입/청산 조건, 전략 파라미터.
- `risk`: 종목당 비중, 주문 가능 수량, 손실/드로다운 차단.
- `broker`: 인증, 조회, 주문, 취소, 정정, 응답 통일.
- `execution`: 주문 생성, 상태 추적, 체결/부분체결, 재시도/예외.
- `portfolio`: 보유 상태, 평균단가, 손익, 잔고, 백테스트 스냅샷.
- `scheduler`: 장전, 장중, 장마감 작업.
- `report`: 주문/체결 리포트, 일일 손익, 장애/경고 알림.
- `logs`: 실행, 주문, 오류, 상태 스냅샷.
- `tests`: 전략, 주문, 리플레이, 모의 시나리오.

## 8. 개발 단계

현재 저장소 구현/테스트 기준이며 실계좌, 외부 API 신청, 실운영 확인은 별도로 본다.

### 8.1 환경 준비

목표: 증권사 계좌/API, 개발 환경, 기본 조회 API.

완료 조건: 현재가, 잔고, 주문 가능 수량 조회와 로그 기록 성공.

완료.

### 8.2 데이터 수집

목표: 유니버스 관리, 일봉/분봉 수집, 로컬 저장 구조.

완료 조건: 지정 ETF/종목 목록 저장, 일봉/분봉 수집, 누락 감지.

완료.

### 8.3 전략 엔진

목표: 단일 전략 구현과 신호 기록.

완료 조건: 인터페이스 고정, 동일 데이터 재현성, 신호 근거 로그.

완료.

### 8.4 백테스트

목표: 전략 성과 보수 검증과 실전 적용성 판단.

필수 반영: 수수료, 세금, 슬리피지, 체결 지연, 과최적화 방지.

평가 지표: CAGR, MDD, 승률, 손익비, 거래 횟수, 평균 보유기간, 월별 일관성.

완료.

남은 리스크/확장:

- 스냅샷은 현금, 평균단가, 실현/평가/총 손익, 총 자산을 저장한다.
- 백테스트는 단일 심볼, 롱온리, 현금 100% 진입/전량 청산 기준이다.
- 체결은 same-bar close 기준이라 주문 지연, 부분체결, 미체결, 정정/취소, 호가 단위, 시장 미세구조를 반영하지 않는다.
- 한국 시장 실수수료/세금 세부 규칙은 추가 검증이 필요하다.
- 멀티 심볼, 자산배분, 현금 배분은 아직 없다.
- 주문 엔진 전에는 위 항목을 재점검하고 실전 전 리플레이/모의투자로 보수 검증한다.

### 8.5 주문 엔진

목표: 조회 중심 코드에서 주문 가능한 코드로 확장하고 주문 정합성을 먼저 해결.

필수 기능: 주문 생성, 정정/취소, 체결 조회, 미체결/부분체결, 중복 주문 방지.

완료.

남은 리스크/확장:

- 한국투자 연동은 주문/정정/취소/체결 동기화를 포함하지만 체결 조회는 일별 주문체결 조회 기반 주문별 집계 스냅샷이다.
- 현재 체결 모델은 주문별 aggregate fill 1건으로 정규화한다. 체결 건 단위 분석에는 실시간 체결통보 또는 상세 체결 API가 필요하다.
- 표준 인터페이스는 국내주식 지정가 주문만 노출한다. 시장가/조건부 주문은 별도 정책이 필요하다.

### 8.6 리스크 관리

목표: 전략보다 먼저 계좌를 보호.

필수 제한: 종목당 비중, 일일 손실, 계좌 드로다운, 동시 보유 수, 주문 횟수, 장 종료 전 미체결 정리.

완료.

남은 리스크/확장:

- 리스크 판단은 `risk/`에 있으나 실주문 차단과 장마감 정리는 orchestration 또는 `execution/` 연동이 필요하다.
- `orders_submitted_today`, `peak_equity`는 스냅샷 입력이므로 운영 경로에서 집계와 주입을 연결해야 한다.

### 8.7 모의투자 및 리플레이

목표: 실계좌 전 운영 시뮬레이션.

완료.

남은 리스크/확장:

- `broker/paper.py`는 지정가 주문 접수/정정/취소/단순 체결을 재현한다.
- `execution/replay.py`는 과거 `Bar` 타임라인에서 `scheduler` 작업을 재실행하고 복구 지점을 제공한다.
- KIS paper read-only smoke와 수동 주문 검증 1회는 통과했고, 로그는 `manual_paper_check_*.log`, `kis_raw_YYYYMMDD.log`로 대조 가능하다.
- 모의 체결은 전량 체결만 지원하고 부분체결, 지연, 시장 미세구조, 호가 반올림은 없다.
- KIS paper `inquire-daily-ccld`는 주문별 `output1`을 비울 수 있어 최종 상태/수량 정합성에 초점을 둔다.
- 실환경 프로세스 복구를 위해 파일 저장/로딩 경로 연결이 필요하다.
- 멀티 심볼, 주문 예약 자금 관리, 리스크 orchestration 결합은 다음 단계다.

### 8.8 소액 실전 운영

목표: 가장 작은 규모로 실전 정합성 확인.

원칙:

- 처음부터 큰 금액을 넣지 않는다.
- 1~2개월은 수익보다 예상대로 동작하는지 본다.
- 이상거래 1회라도 즉시 원인 분석 후 재개한다.

현재 요약:

- 한 운영 사이클에서 전략 입력 바 수집, 신호 계산, 리스크 검증, 주문 제출, 주문/체결 알림, 주문 상태 파일 저장이 가능하다.
- `scheduler`는 장 시작/장중/장마감 슬롯과 중복 방지 상태를 제공한다.
- 운영 리포트와 주간 리뷰 모델/렌더러는 준비되어 있다.
- 파일 notifier와 Telegram notifier가 연결되어 있다.

#### 8.8.1 24시간 운영 오케스트레이션

장시간 프로세스가 다음 실행 시각을 계산하고 재시작 후 중복 없이 이어가야 한다. 현재 safe stop 경로도 daily report와 다음 거래일 준비 산출물을 남긴다.

완료.

#### 8.8.2 장 시작 전 준비

`.env` 설정, 08:00 장전 슬롯, 전략 바 최신화, read-only smoke, inspection report, `market_open` job이 운영 루프에 연결되어 있다. 장전 준비는 인증, 잔고/주문가능수량, 전일 오류, 비상 정지, 대상 종목, 전략 파라미터를 점검하고 요약을 notifier/Telegram으로 발행한다.

완료.

#### 8.8.3 장중 전략 기준 실시간 매매

장중 경로는 분봉 기반 저빈도 전략 실행, 주문 제출, pending 주문 재조회, 부분체결 반영, 장마감 전 미체결 정리까지 확장되었다. 실시간은 websocket 이벤트보다 REST 분봉 조회 주기 실행에 가깝다. `paper`는 내부 시뮬레이터가 기본이며 KIS paper 실주문은 `AUTOTRADE_PAPER_TRADING_MODE=broker`로 전환한다.

- [x] 분봉/일봉 기반 전략 신호 계산
- [x] 리스크 검증 후 주문 제출
- [x] 동일 방향 진행 중 주문 중복 방지
- [x] 주문 상태 파일 저장
- [x] pending 주문 주기 재조회 및 후속 체결 반영
- [x] 부분체결/미체결 기반 후속 주문 정책
- [x] 장중 주문 횟수, 손실 한도, 드로다운 상태값 집계
- [ ] 이벤트 기반 체결통보 또는 더 촘촘한 주문 상태 추적
- [x] 장 종료 직전 신규 진입 제한과 미체결 정리

#### 8.8.4 장 종료 후 결과 정리

`market_close` job은 당일 job 결과, 주문/체결/잔고를 모아 daily report와 inspection report를 만들고 다음 거래일 준비 항목을 남긴다. Safe stop도 같은 정리 경로를 재사용한다.

완료.

#### 8.8.5 주간 리포트 및 평가

주간 리뷰 모델/렌더러/파일 출력이 준비되어 있고, 일일 실행/점검 리포트 누적 경로를 수집해 실패 일자, 점검 누락, 반복 오류를 집계한다.

완료.

#### 8.8.6 텔레그램 알림 및 리포트

주문/체결/일일 리포트 알림 모델은 notifier protocol 위에 있으며 파일/Telegram fan-out, bot token/chat id, 429/5xx/네트워크 재시도, 장문 분할, warning/error 채널 분리를 지원한다.

완료.

#### 8.8.7 주간 종목 추천 파이프라인
