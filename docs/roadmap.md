# 국내 ETF 및 단일 종목 현물 자동매매 로드맵 (개인 개발자용)

## 1. 문서 목적

이 문서는 **국내 ETF 및 단일 종목 현물 자동매매 시스템**을 개인 개발자 관점에서 설계·구현·검증·운영하기 위한 기준 문서이다.  
첫 목표는 “고수익 전략”이 아니라 **안전하게 동작하는 자동매매 MVP를 만드는 것**이다.


이 문서는 아래 상황을 전제로 한다.

- 대상 시장: 국내 ETF 및 국내 주식 단일 종목 현물
- 개발 주체: 개인 개발자
- 운영 방식: 저빈도 자동매매
- 초기 목표: 소액 실전 가능한 안정적인 1차 버전 구축
- 우선순위: 전략 복잡도보다 시스템 안정성, 리스크 통제, 운영 가능성

---

## 2. 기본 방향

### 핵심 원칙

1. 처음부터 복잡한 전략을 만들지 않는다.
2. 레버리지/인버스/선물/초단타는 초기 범위에서 제외한다.
3. 전략 코드와 증권사 API 코드는 분리한다.
4. 백테스트 → 모의투자 → 소액 실전 → 점진 증액 순서로 진행한다.
5. 수익률보다 **장애 대응, 주문 정합성, 리스크 제한**을 먼저 완성한다.

### 초기 권장 조합

- 상품: 국내 ETF 및 주식 단일 종목 포함
- 거래 시간: 정규장만
- 매매 빈도: 하루 1회 또는 30분~60분 단위
- 보유 종목 수: 최대 1~3개
- 주문 방식: 지정가 위주
- 전략 수: 1개만 먼저 운영

## 5. 매매 유니버스 선정 기준

초기에는 ETF와 국내 주식 단일 종목을 합쳐 3~10개 정도만 관리한다.

### 공통 선정 기준

- 거래대금이 충분할 것
- 유동성이 좋을 것
- 구조 또는 사업 내용이 이해하기 쉬울 것
- 운영 중 선정 사유를 설명할 수 있을 것
- 특정 이벤트 하나에 포트폴리오가 과도하게 흔들리지 않을 것

### ETF 추가 기준

- 광범위 지수형일 것
- 괴리율/NAV 이슈가 과도하지 않을 것

### 국내 주식 추가 기준

- 시가총액과 유동성이 충분할 것
- 단기 테마주보다 업종 대표주 또는 실적 기반 종목일 것
- 단일 공시/수급 이벤트에 과도하게 좌우되지 않을 것

### 초기 제외 기준

- 테마형 ETF
- 거래량이 적은 ETF 또는 종목
- 급격한 변동성에만 의존하는 ETF 또는 종목
- 레버리지/인버스 ETF
- 관리종목, 투자주의/경고/위험 종목

### 유니버스 예시 방향

- 대형지수형 ETF
- 기술/성장 대형지수형 ETF
- 채권형 또는 현금대용 성격 ETF 일부
- 코스피 대형주 또는 업종 대표주
- 실적과 유동성이 안정적인 우량주

## 6. 전략 방향

## 6.1 첫 전략의 조건

첫 전략은 반드시 아래 조건을 만족해야 한다.

- 설명 가능해야 한다.
- 룰이 단순해야 한다.
- 백테스트가 쉬워야 한다.
- 실전에서 왜 매수/매도했는지 추적 가능해야 한다.

## 6.2 권장 전략 유형

### A. 일봉 추세추종
- 이동평균선 기반 진입/이탈
- 장점: 구조 단순, 검증 쉬움
- 단점: 횡보장에서 성과 저하 가능

### B. 30분봉 저빈도 추세 전략
- 장중 30분 간격으로 신호 확인
- 장점: 일봉보다 반응 빠름
- 단점: 체결/슬리피지 관리 필요

### C. 변동성 필터 + 현금 비중 조절
- 시장이 불안정하면 현금 비중 확대
- 장점: 낙폭 관리에 유리
- 단점: 강한 상승장에서 기회손실 가능

## 6.3 초기 비추천 전략

- 조건이 너무 많은 복합 전략
- 분 단위 초단타 전략
- 뉴스/텍스트 기반 전략
- AI 모델에 강하게 의존하는 전략

## 7. 시스템 아키텍처

처음부터 모듈 분리를 한다.

```text
AutoTrade/
├─ config/
├─ data/
├─ strategy/
├─ risk/
├─ broker/
├─ execution/
├─ portfolio/
├─ scheduler/
├─ report/
├─ logs/
└─ tests/
```

### 모듈 역할

#### config
- 환경설정
- 계좌/장운영/종목 리스트
- API 키 및 운영 파라미터

#### data
- 시세 수집
- 대상 종목 목록 관리
- 일봉/분봉 저장
- 데이터 정합성 체크

#### strategy
- 신호 계산
- 진입/청산 조건 관리
- 전략 파라미터 분리

#### risk
- 종목당 최대 비중
- 주문 가능 수량 계산
- 일일 손실 상한
- 최대 드로다운 차단

#### broker
- 증권사 API 래퍼
- 인증, 조회, 주문, 취소, 정정
- 브로커별 응답 형식 통일

#### execution
- 주문 생성
- 주문 상태 추적
- 체결/부분체결 처리
- 재시도 및 예외 처리

#### portfolio
- 보유 종목 상태 관리
- 평균단가, 손익, 잔고 추적
- 백테스트 기준 현금, 보유 수량, 평균단가, 실현/평가 손익 스냅샷 생성
- 실전 상태와 내부 상태 동기화

#### scheduler
- 장 시작 전 준비
- 장중 주기 실행
- 장 종료 후 정리 작업

#### report
- 주문/체결 리포트
- 일일 손익 요약
- 장애 및 경고 알림

#### logs
- 실행 로그
- 주문 로그
- 오류 로그
- 상태 스냅샷

#### tests
- 전략 단위 테스트
- 주문 로직 테스트
- 리플레이 테스트
- 모의 시나리오 테스트

### 현재 구현 기준 남은 리스크 / 추후 확장 포인트

- 현재 백테스트 스냅샷은 현금, 평균단가, 실현 손익, 평가 손익, 총 손익, 총 자산을 저장한다.
- 현재 백테스트는 단일 심볼 기준으로만 동작한다.
- 현재 포지션 모델은 롱온리이며 현금 100% 진입, 전량 청산 기준이다.
- 현재 체결 모델은 same-bar close 체결 기준이라 실제 주문 지연을 반영하지 않는다.
- 부분체결, 미체결, 정정/취소 같은 주문 상태 변화는 아직 반영하지 않는다.
- 호가 단위 반올림과 실제 시장 미세구조는 아직 체결가 계산에 반영하지 않는다.
- 한국 시장 기준의 실수수료/세금 세부 규칙은 단순 비율 모델에서 추가 검증이 필요하다.
- 멀티 심볼 포트폴리오, 자산배분, 현금 배분 로직은 아직 포함하지 않는다.
- 주문 엔진 개발 전에는 위 항목을 우선순위로 다시 점검하고, 실전 전에는 리플레이 테스트와 모의투자로 보수적으로 검증한다.

### 현재 구현 기준 남은 리스크 / 추후 확장 포인트

- 한국투자 브로커 연동은 주문/정정/취소/체결 동기화까지 포함하지만, 체결 조회는 일별 주문체결 조회 기반의 주문별 집계 스냅샷이다.
- 현재 체결 모델은 주문별 aggregate fill 1건으로 정규화하므로, 개별 체결 건 단위 분석이 필요하면 실시간 체결통보 또는 상세 체결 API를 추가로 연결해야 한다.
- 국내주식 지정가 주문만 표준 인터페이스로 노출하며, 시장가/조건부 주문 정책은 주문 정책 단계에서 별도 설계가 필요하다.

- `broker/paper.py`는 결정적 모의 브로커로서 지정가 주문의 접수, 정정, 취소, 단순 체결을 재현한다.

- 한국투자 paper 계좌 기준 read-only smoke와 수동 주문 검증 1회는 통과했고, 제출/정정/취소/체결 뒤 잔고 및 raw HTTP 로그 대조까지 확인했다.
- paper 검증 로그는 `manual_paper_check_*.log`와 `kis_raw_YYYYMMDD.log` 기준으로 대조 가능하다.
- 현재 모의 체결은 전량 체결만 지원하며 부분체결, 체결 지연, 장내 미세구조, 실제 호가 단위 반올림은 반영하지 않는다.
- 한국투자 paper `inquire-daily-ccld`는 주문별 `output1`을 비워줄 수 있어서, 현재는 주문별 체결 상세보다 최종 상태/수량 정합성 확인에 초점을 둔다.
- 재시작 복구는 replay 세션 상태와 로그 엔트리 기준으로 검증되지만, 실환경 프로세스 복구를 위해서는 파일 저장/로딩 경로를 별도로 연결해야 한다.

## 8.8 8단계 - 소액 실전 운영

### 목표
- 가장 작은 규모로 실전 정합성 확인

### 원칙
- 처음부터 큰 금액 넣지 않는다.
- 1~2개월은 “수익”보다 “예상대로 동작하는지”를 본다.
- 이상거래 1회라도 나오면 즉시 원인 분석 후 재개한다.

### 현재 구현 기준 요약
- 현재 저장소는 전략 입력 바 수집, 전략 신호 계산, 리스크 검증, 주문 제출, 주문/체결 알림 생성, 주문 상태 파일 저장까지 한 번의 운영 사이클로 실행할 수 있다.
- 현재 `scheduler`는 장 시작, 장중, 장마감 슬롯 계산과 중복 실행 방지 상태를 제공하지만, 이를 실제 프로세스로 계속 돌리는 상위 runner/daemon은 아직 없다.
- 현재 운영 리포트와 주간 리뷰는 데이터 모델과 렌더러가 준비되어 있지만, 실제 실행 로그를 자동 집계해 일일/주간 산출물로 연결하는 orchestration은 아직 없다.
- 현재 알림 채널은 파일 기반 notifier만 구현되어 있으며, 텔레그램 같은 외부 알림 채널은 아직 연결하지 않았다.

- 전략이 자동으로 신호를 생성한다.
- 주문이 의도대로 나간다.
- 체결 상태가 내부 상태와 일치한다.
- 장애 발생 시 복구 가능하다.

## 16. 한 줄 결론

첫 자동매매 프로젝트는  
**“국내 ETF/단일 종목 현물 + 저빈도 + 단일 전략 + 보수적 주문 정책 + 강한 리스크 통제”**  
이 조합으로 시작하는 것이 가장 현실적이다.

수익률보다 먼저 완성해야 할 것은  
**주문 안정성, 상태 정합성, 로그 추적성, 비상 정지 가능성**이다.
