# 국내 ETF 및 단일 종목 현물 자동매매 로드맵 (개인 개발자용)

## 1. 문서 목적

이 문서는 **국내 ETF 및 단일 종목 현물 자동매매 시스템**을 개인 개발자 관점에서 설계·구현·검증·운영하기 위한 기준 문서이다.  
첫 목표는 “고수익 전략”이 아니라 **안전하게 동작하는 자동매매 MVP를 만드는 것**이다.


이 문서는 아래 상황을 전제로 한다.

- 대상 시장: 국내 ETF 및 국내 주식 단일 종목 현물
- 개발 주체: 개인 개발자
- 운영 방식: 저빈도 자동매매
- 초기 목표: 소액 실전 가능한 안정적인 1차 버전 구축
- 우선순위: 전략 복잡도보다 시스템 안정성, 리스크 통제, 운영 가능성

---

## 2. 기본 방향

### 핵심 원칙

1. 처음부터 복잡한 전략을 만들지 않는다.
2. 레버리지/인버스/선물/초단타는 초기 범위에서 제외한다.
3. 전략 코드와 증권사 API 코드는 분리한다.
4. 백테스트 → 모의투자 → 소액 실전 → 점진 증액 순서로 진행한다.
5. 수익률보다 **장애 대응, 주문 정합성, 리스크 제한**을 먼저 완성한다.

### 초기 권장 조합

- 상품: 국내 ETF 및 주식 단일 종목 포함
- 거래 시간: 정규장만
- 매매 빈도: 하루 1회 또는 30분~60분 단위
- 보유 종목 수: 최대 1~3개
- 주문 방식: 지정가 위주
- 전략 수: 1개만 먼저 운영

## 5. 매매 유니버스 선정 기준

초기에는 ETF와 국내 주식 단일 종목을 합쳐 3~10개 정도만 관리한다.

### 공통 선정 기준

- 거래대금이 충분할 것
- 유동성이 좋을 것
- 구조 또는 사업 내용이 이해하기 쉬울 것
- 운영 중 선정 사유를 설명할 수 있을 것
- 특정 이벤트 하나에 포트폴리오가 과도하게 흔들리지 않을 것

### ETF 추가 기준

- 광범위 지수형일 것
- 괴리율/NAV 이슈가 과도하지 않을 것

### 국내 주식 추가 기준

- 시가총액과 유동성이 충분할 것
- 단기 테마주보다 업종 대표주 또는 실적 기반 종목일 것
- 단일 공시/수급 이벤트에 과도하게 좌우되지 않을 것

### 초기 제외 기준

- 테마형 ETF
- 거래량이 적은 ETF 또는 종목
- 급격한 변동성에만 의존하는 ETF 또는 종목
- 레버리지/인버스 ETF
- 관리종목, 투자주의/경고/위험 종목

### 유니버스 예시 방향

- 대형지수형 ETF
- 기술/성장 대형지수형 ETF
- 채권형 또는 현금대용 성격 ETF 일부
- 코스피 대형주 또는 업종 대표주
- 실적과 유동성이 안정적인 우량주

## 6. 전략 방향

## 6.1 첫 전략의 조건

첫 전략은 반드시 아래 조건을 만족해야 한다.

- 설명 가능해야 한다.
- 룰이 단순해야 한다.
- 백테스트가 쉬워야 한다.
- 실전에서 왜 매수/매도했는지 추적 가능해야 한다.

## 6.2 권장 전략 유형

### A. 일봉 추세추종
- 이동평균선 기반 진입/이탈
- 장점: 구조 단순, 검증 쉬움
- 단점: 횡보장에서 성과 저하 가능

### B. 30분봉 저빈도 추세 전략
- 장중 30분 간격으로 신호 확인
- 장점: 일봉보다 반응 빠름
- 단점: 체결/슬리피지 관리 필요

### C. 변동성 필터 + 현금 비중 조절
- 시장이 불안정하면 현금 비중 확대
- 장점: 낙폭 관리에 유리
- 단점: 강한 상승장에서 기회손실 가능

## 6.3 초기 비추천 전략

- 조건이 너무 많은 복합 전략
- 분 단위 초단타 전략
- 뉴스/텍스트 기반 전략
- AI 모델에 강하게 의존하는 전략

## 7. 시스템 아키텍처

처음부터 모듈 분리를 한다.

```text
AutoTrade/
├─ config/
├─ data/
├─ strategy/
├─ risk/
├─ broker/
├─ execution/
├─ portfolio/
├─ scheduler/
├─ report/
├─ logs/
└─ tests/
```

### 모듈 역할

#### config
- 환경설정
- 계좌/장운영/종목 리스트
- API 키 및 운영 파라미터

#### data
- 시세 수집
- 대상 종목 목록 관리
- 일봉/분봉 저장
- 데이터 정합성 체크

#### strategy
- 신호 계산
- 진입/청산 조건 관리
- 전략 파라미터 분리

#### risk
- 종목당 최대 비중
- 주문 가능 수량 계산
- 일일 손실 상한
- 최대 드로다운 차단

#### broker
- 증권사 API 래퍼
- 인증, 조회, 주문, 취소, 정정
- 브로커별 응답 형식 통일

#### execution
- 주문 생성
- 주문 상태 추적
- 체결/부분체결 처리
- 재시도 및 예외 처리

#### portfolio
- 보유 종목 상태 관리
- 평균단가, 손익, 잔고 추적
- 백테스트 기준 현금, 보유 수량, 평균단가, 실현/평가 손익 스냅샷 생성
- 실전 상태와 내부 상태 동기화

#### scheduler
- 장 시작 전 준비
- 장중 주기 실행
- 장 종료 후 정리 작업

#### report
- 주문/체결 리포트
- 일일 손익 요약
- 장애 및 경고 알림

#### logs
- 실행 로그
- 주문 로그
- 오류 로그
- 상태 스냅샷

#### tests
- 전략 단위 테스트
- 주문 로직 테스트
- 리플레이 테스트
- 모의 시나리오 테스트

- 현재 포지션 모델은 롱온리이며 현금 100% 진입, 전량 청산 기준이다.
- 현재 체결 모델은 same-bar close 체결 기준이라 실제 주문 지연을 반영하지 않는다.
- 부분체결, 미체결, 정정/취소 같은 주문 상태 변화는 아직 반영하지 않는다.
- 호가 단위 반올림과 실제 시장 미세구조는 아직 체결가 계산에 반영하지 않는다.
- 한국 시장 기준의 실수수료/세금 세부 규칙은 단순 비율 모델에서 추가 검증이 필요하다.
- 멀티 심볼 포트폴리오, 자산배분, 현금 배분 로직은 아직 포함하지 않는다.
- 주문 엔진 개발 전에는 위 항목을 우선순위로 다시 점검하고, 실전 전에는 리플레이 테스트와 모의투자로 보수적으로 검증한다.

## 9. 주문 정책

초기 버전 주문 정책은 최대한 보수적으로 간다.

### 권장 정책

- 지정가 주문 우선
- 장 시작 직후 무리한 진입 금지
- 장 종료 직전 신규 진입 제한
- 장 종료 전 미체결 정리
- 동일 종목 중복 진입 금지

### 비추천 정책

- 손절/청산 로직 없는 진입
- 체결 확인 없는 후속 주문
- 내부 상태 저장 없이 실시간 상태만 신뢰

## 12. 개인 개발자 기준 권장 운영 스펙

### 1차 운영 스펙
- 대상 종목 수: 3~5개
- 동시 보유: 최대 1~2개
- 매매 빈도: 하루 1회 또는 30분봉
- 주문: 지정가 위주
- 전략: 1개
- 운영 시간: 정규장만
- 서버: 개발은 로컬 가능, 실전은 항상 켜져 있는 환경 권장
- 알림: 주문/체결/오류/일일 요약 필수

### 1차 성공 기준
아래가 되면 1차 성공으로 본다.

- 전략이 자동으로 신호를 생성한다.
- 주문이 의도대로 나간다.
- 체결 상태가 내부 상태와 일치한다.
- 장애 발생 시 복구 가능하다.
- 일일 손익과 로그를 확인할 수 있다.
