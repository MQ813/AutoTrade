# Coding Standards

## 문서 목적

이 문서는 AutoTrade 코드 작성 시 지켜야 할 기본 규칙을 정의합니다.
목표는 취향 통일이 아니라 정확성, 재현성, 테스트 용이성, 안전한 리팩터링입니다.

## 기본 원칙

- 작은 변경으로 문제를 해결합니다.
- 하나의 함수와 모듈은 하나의 책임에 집중합니다.
- 가능한 한 순수 함수를 우선합니다.
- 숨은 상태보다 명시적인 입력과 반환값을 선호합니다.
- 구현 편의보다 검증 가능성과 운영 안정성을 우선합니다.
- 리팩터링과 동작 변경은 한 패치에 섞지 않습니다.

## 모듈 경계

- `config`: 설정 로드, 환경변수 해석, 운영 파라미터 관리
- `data`: 시세 수집, 정규화, 저장, 데이터 검증
- `strategy`: 신호 계산과 전략 규칙
- `risk`: 포지션 제한, 손실 제한, 주문 가능 범위 계산
- `broker`: 증권사 API 래퍼와 외부 응답 정규화
- `execution`: 주문 생성, 상태 전이, 재시도 처리
- `portfolio`: 보유 상태와 내부 포지션 모델 관리
- `scheduler`: 실행 순서 조합과 작업 스케줄링
- `report`: 리포트와 운영 출력 생성
- `common`: 여러 모듈이 공유하는 작고 안정적인 유틸리티

아래 규칙을 항상 지킵니다.

- 파싱, 변환, I/O를 한 함수에 섞지 않습니다.
- CLI 또는 엔트리포인트에는 비즈니스 로직을 넣지 않습니다.
- 공유 유틸리티는 `src/autotrade/common/`에 둡니다.
- 다른 레이어의 내부 구현을 직접 가져오지 말고 공개된 인터페이스를 통해 연결합니다.

## 함수와 클래스

- 함수는 가능하면 입력을 받아 값을 반환하는 형태로 작성합니다.
- 함수가 길어지면 단계별 작은 함수로 분리합니다.
- 이름은 의도를 설명해야 합니다.
- Boolean 플래그로 여러 동작을 전환하는 함수는 지양합니다.
- 하나의 함수에서 여러 수준의 추상화를 섞지 않습니다.
- 클래스는 상태와 책임이 명확할 때만 도입합니다.
- 단순 데이터 묶음은 복잡한 클래스보다 `dataclass` 같은 단순 구조를 우선합니다.

## 타입과 데이터 모델

- 공개 함수와 모듈 경계 함수에는 타입 힌트를 작성합니다.
- 데이터는 가능한 한 명시적인 구조로 표현합니다.
- `dict[str, Any]` 같은 넓은 타입은 경계 구간에 한정합니다.
- 브로커 응답, 설정, 리포트 데이터는 내부 표준 형태로 빠르게 정규화합니다.
- `None`이 의미 있는 값이라면 문서화하거나 이름으로 드러냅니다.

## 상태와 시간

- 시간, 랜덤성, 외부 응답에 의존하는 값은 주입 가능하게 설계합니다.
- 전역 상태와 암묵적 캐시는 지양합니다.
- 재실행 시 같은 입력이면 같은 결과를 내는 코드를 우선합니다.
- 운영 상태 변경은 한 곳에서만 수행되도록 경계를 명확히 둡니다.

## 예외와 로깅

- 예외를 무시하지 않습니다.
- 외부 시스템 오류와 도메인 규칙 위반은 구분해서 다룹니다.
- 낮은 레벨에서 의미 없는 로그를 남기기보다 상위 경계에서 맥락 있는 로그를 남깁니다.
- 순수 로직은 출력보다 반환값과 예외로 의사를 표현합니다.
- 재시도는 `execution` 또는 외부 I/O 경계에서만 다룹니다.

## 스타일 규칙

- 기본 포맷은 `ruff format .` 결과를 따릅니다.
- 린트는 `ruff check .` 기준을 통과해야 합니다.
- 불필요한 축약어보다 이해하기 쉬운 이름을 사용합니다.
- 상수는 의미 있는 이름으로 분리합니다.
- 매직 넘버는 거래 규칙이나 운영 제한과 연결될 때 이름을 붙입니다.
- 주석은 코드가 무엇을 하는지보다 왜 그렇게 해야 하는지를 설명할 때만 추가합니다.

## 임포트와 의존성

- 표준 라이브러리, 서드파티, 로컬 임포트 순서를 유지합니다.
- 순환 참조를 만들지 않습니다.
- 새 의존성 추가는 마지막 수단으로 생각합니다.
- 테스트 편의를 위해 구현 세부사항보다 인터페이스에 의존합니다.

## 입출력과 설정

- 파일, 네트워크, 환경변수 접근은 경계 모듈에 모읍니다.
- 내부 로직은 가능한 한 파싱된 값과 표준화된 객체만 다룹니다.
- 사용자 데이터나 운영 산출물을 덮어쓸 때는 안전한 경로를 먼저 고려합니다.
- 대량 변환은 수동 편집보다 결정적인 스크립트를 우선합니다.

## 테스트 친화성

- 새 로직은 테스트 가능한 형태로 먼저 분리합니다.
- 전략, 리스크, 주문 상태 전이는 순수 로직 단위 테스트로 고정합니다.
- 테스트는 결정적이어야 하며 현재 시각, 랜덤 값, 외부 API에 직접 의존하지 않습니다.
- 픽스처는 최소한으로 유지하고, 테스트 의도가 이름에서 드러나야 합니다.
- 버그 수정 시 가능하면 재현 테스트를 함께 추가합니다.

## 변경 단위

- 한 커밋 또는 한 패치는 하나의 의도를 가져야 합니다.
- 동작 변경이 없다면 이름 변경과 구조 정리는 따로 분리합니다.
- 큰 변경 전에는 먼저 경계와 데이터 모델을 고정합니다.
- 구현 중간 단계라도 프로그램이 깨진 상태로 오래 두지 않습니다.

## 검증 기준

Python 코드 변경 시 아래 순서를 기본으로 합니다.

1. 관련 범위의 가장 가까운 테스트 실행
2. `ruff check .`
3. `mypy src/`
4. `pytest tests/unit -q`

문서만 수정한 경우에는 별도 자동 검증이 없을 수 있지만, 변경 범위와 검증 여부를 작업 요약에 명시합니다.
